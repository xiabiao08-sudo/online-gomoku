# 在线联机五子棋设计文档

日期：2026-07-05

## 目标

设计并实现一个公网可访问的网页好友房五子棋 MVP。用户不需要注册账号，只要打开网页、输入昵称、创建或加入房间，就能和好友实时对弈。

第一版强调完整闭环：创建房间、分享链接、双人进入、实时落子、判胜、断线重连、再来一局。界面采用传统棋盘风：暖色棋盘底、细网格、黑白棋子、简洁边框和低干扰状态提示。

## 已确认决策

- 平台：网页游戏，支持电脑和手机浏览器。
- 联机方式：好友房联机。
- 身份方式：不登录，昵称进入房间。
- 技术路线：React + Vite 前端，Node.js + Express + Socket.IO 后端。
- 部署目标：公网可访问演示版。
- 棋盘尺寸：15 x 15。
- 规则：自由五子棋，横、竖、两条斜线任意五连即胜，不做禁手。
- 计时：第一版不加计时。
- 悔棋：第一版不支持悔棋。
- 重连：短暂断线可重连，房间保留一段时间。
- 数据存储：第一版不接数据库，房间状态保存在服务端内存。

## 非目标

第一版不做账号系统、排行榜、历史战绩、观战、聊天、AI 对手、匹配队列、禁手规则、计时器、悔棋协商、皮肤商城或长期房间保存。

这些功能可以作为后续版本扩展，不能阻塞 MVP。

## 用户流程

### 创建房间

1. 用户打开首页。
2. 输入昵称。
3. 点击“创建房间”。
4. 服务端创建房间，用户进入房间页并分配为黑棋。
5. 页面显示房间号和可复制链接。
6. 用户把链接发给好友。

### 加入房间

1. 好友打开房间链接，或在首页输入房间号。
2. 输入昵称。
3. 服务端校验房间存在、未过期、未满员。
4. 好友进入房间并分配为白棋。
5. 两人到齐后自动开始，黑棋先手。

### 对局

1. 当前回合玩家点击棋盘交叉点。
2. 前端发送落子请求。
3. 服务端校验并写入棋盘。
4. 服务端检查是否五连胜。
5. 服务端广播完整房间状态。
6. 双方界面同步更新。

### 结束与再来一局

1. 任一方向形成五连后，服务端标记棋局结束。
2. 前端高亮获胜五连线并显示胜负弹窗。
3. 任一方可以点击“再来一局”发起下一局。
4. 为避免一方误触，推荐第一版采用“双方准备后开始下一局”。

### 断线与重连

1. 玩家刷新、关闭标签页或网络短断时，服务端将该玩家标记为离线。
2. 对方看到“对方离线，等待重连”。
3. 本地浏览器保存 `roomId` 和 `playerToken`。
4. 玩家重新打开房间链接后，前端带 token 请求恢复身份。
5. token 匹配时，服务端把新 socket 绑定到原玩家并恢复房间状态。
6. 无人在线超过 30 分钟后，房间可被清理。

## 页面设计

### 首页 `/`

首页提供最少入口：

- 昵称输入框。
- 创建房间按钮。
- 房间号输入框。
- 加入房间按钮。
- 如果本地存在未结束房间 token，可显示“继续对局”。

首页不做营销页，不放复杂介绍。重点是快速进入游戏。

首页按钮和输入框必须满足游戏 UI 的触控下限：可点击区域不小于 48 x 48 CSS px，按钮之间至少保留 8px 间距。键盘用户可以用 Tab 顺序完成昵称输入、创建房间、输入房间号、加入房间的完整流程，焦点样式不能只靠颜色变化。

### 房间页 `/room/:roomId`

房间页是核心界面：

- 15 x 15 棋盘。
- 黑白双方昵称、棋色、在线状态。
- 当前回合提示。
- 房间号和复制链接按钮。
- 错误或等待提示。
- 对局结束弹窗。
- 再来一局/准备下一局按钮。

桌面端采用棋盘居中、状态栏在右侧或上方。手机端棋盘优先占据主要宽度，状态信息压缩到棋盘上下方，避免挡住落子区域。

房间页采用低干扰 HUD。房间号、复制链接、玩家在线状态和当前回合提示都应位于棋盘安全区外，不覆盖落子区域。移动端横竖屏都保留页面边缘安全边距，避免状态文字或按钮贴到屏幕边缘。

### 房间失效页

用于处理：

- 房间不存在。
- 房间已过期。
- 房间已满。
- token 无效且无法恢复身份。

页面提供返回首页和重新创建房间入口。

## 视觉方向

整体风格参考传统棋盘：

- 棋盘底色：暖木色或淡纸色。
- 网格：细线、低饱和深色。
- 棋子：黑白实体圆棋子，带轻微阴影。
- 最近落子：用小圆点或细描边标记。
- 获胜五连：用低干扰高亮圈或连线强调。
- 状态信息：简洁，不遮挡棋盘。

设计重点是安静、耐看、易点准，而不是强竞技、强装饰。

视觉系统应有明确的传统棋局气质，而不是通用后台工具风。建议方向：

- 标题和房间号使用有纸本文气的中文字体栈，例如 `Noto Serif SC`、`LXGW WenKai`、`Songti SC`，再回退到系统字体。
- 棋盘使用轻微纸纹或木纹质感，但不使用会影响网格辨识的强纹理。
- 黑白棋子使用实体阴影和细高光表达层次，避免过度 3D 或花哨发光。
- 状态色只作辅助，必须配合文字、形状或图标。例如当前回合不能只用颜色区分，还要显示“黑棋落子”“白棋落子”。
- 动效只用于沟通状态：落子轻微缩放、胜利五连高亮、断线提示淡入。常规 UI 动效控制在 200ms 以内，并尊重 `prefers-reduced-motion`。

## 游戏 UI 交互反馈

五子棋的核心反馈来自“我能不能下、下在哪里、刚刚发生了什么”。第一版需要设计这些状态：

### 棋盘状态

- 可落子：当前玩家回合时，空交叉点可显示半透明预落子。
- 不可落子：非当前玩家回合、棋局未开始、对方离线或棋局结束时，棋盘保持可读但不响应落子。
- 非法落子：点击已有棋子、越界或非自己回合时，不移动棋子，只显示短提示。
- 最近落子：最后一步用小圆点或细描边标记，方便玩家回看。
- 获胜五连：高亮五颗棋子和方向线，胜负弹窗出现后仍能看到棋盘结果。

### 触控与点击命中

15 x 15 棋盘在手机上容易误触。实现时不要只把棋子本身作为点击目标，而应以交叉点为中心建立隐形命中区域。棋盘在常见手机宽度下应尽量接近可用宽度，保证单个格距足够点击；如果屏幕太窄，允许棋盘横向保持比例并通过布局压缩状态栏，而不是缩小到无法点准。

### 状态提示

状态提示只显示当前最重要的信息：

1. 等待好友加入。
2. 轮到你落子。
3. 等待对方落子。
4. 对方离线，等待重连。
5. 黑棋/白棋获胜。
6. 双方准备后开始下一局。

提示不能堆叠成通知流。新错误可以短暂覆盖次要提示，但不能遮挡棋盘中心。

## 可访问性与响应式要求

第一版虽然是 MVP，也必须避免常见游戏 UI 问题：

- 安全区：静态文字和按钮不要贴屏幕边缘，桌面和移动端都保留清晰边距。
- 字号：正文和状态文字不小于 16px，关键回合提示建议 20px 以上。
- 触控目标：按钮、复制链接、再来一局、返回首页等交互区不小于 48 x 48 CSS px。
- 焦点：所有按钮和输入框都有明显 `:focus-visible` 样式。
- 色彩：黑白棋子天然可辨，但当前回合、在线/离线、胜负状态不能只靠颜色表达。
- 动效：支持系统减少动态效果设置；禁用后仍能通过静态标记理解最近落子和胜利线。
- 文本可读性：状态文字放在稳定背景或半透明纸色面板上，不直接压在复杂纹理上。
- 视口：至少验证 360px 宽手机、768px 平板、1440px 桌面三类宽度。

## 技术架构

项目采用单体实时网页应用：

- 前端：React + Vite。
- 后端：Node.js + Express。
- 实时通信：Socket.IO。
- 状态存储：服务端内存。
- 部署：一个服务同时提供静态资源和 WebSocket 服务。

前端只负责展示和发送用户意图，不负责最终裁判。所有落子合法性、回合顺序和胜负判定都由服务端决定。

## 服务端数据模型

```text
Room {
  id: string
  players: {
    black?: Player
    white?: Player
  }
  board: Cell[15][15]
  currentTurn: "black" | "white"
  status: "waiting" | "playing" | "finished"
  winner?: "black" | "white"
  winningLine?: Point[]
  lastMove?: Point
  restartReady: {
    black: boolean
    white: boolean
  }
  createdAt: number
  updatedAt: number
  expiresAt?: number
}

Player {
  id: string
  token: string
  nickname: string
  color: "black" | "white"
  socketId?: string
  online: boolean
  joinedAt: number
  lastSeenAt: number
}

Cell = null | "black" | "white"

Point {
  x: number
  y: number
}
```

## Socket 事件

| 事件 | 方向 | 说明 |
| --- | --- | --- |
| `room:create` | client -> server | 提交昵称，创建房间 |
| `room:join` | client -> server | 提交房间号、昵称或重连 token，加入房间 |
| `room:state` | server -> client | 广播完整房间状态 |
| `game:placeStone` | client -> server | 请求落子 |
| `game:restartReady` | client -> server | 标记自己准备下一局 |
| `player:leave` | client -> server | 主动离开房间 |
| `error:message` | server -> client | 返回房间满、过期、非法落子等错误 |

服务端广播的 `room:state` 应该是前端渲染所需的完整状态，避免前端靠事件补丁拼接出错。

## 落子校验

服务端收到 `game:placeStone` 后按顺序校验：

1. 房间存在。
2. 玩家 token 有效。
3. 玩家属于该房间。
4. 房间状态是 `playing`。
5. 两名玩家都在线。
6. 当前轮到该玩家。
7. 坐标在 15 x 15 范围内。
8. 目标位置为空。

校验通过后写入棋盘、记录最近落子、检查胜负。如果未结束，则切换回合。

## 判胜算法

从最后一步落子出发检查四个方向：

- 横向 `(1, 0)`。
- 纵向 `(0, 1)`。
- 左上到右下 `(1, 1)`。
- 右上到左下 `(1, -1)`。

每个方向向正反两边统计连续同色棋子数量。总数大于或等于 5 即胜。自由五子棋不处理禁手，也不因为长连取消胜利。

## 错误处理

需要给用户明确但简短的错误提示：

- 昵称为空。
- 房间号不存在。
- 房间已过期。
- 房间已满。
- 不是你的回合。
- 该位置已有棋子。
- 对方离线，暂时不能落子。
- 连接中断，正在重连。

错误不应导致页面崩溃。前端收到服务端错误后显示提示，并等待下一次 `room:state` 同步。

错误提示要避免游戏 UI 的两个问题：第一，不要弹出会遮挡棋盘中心的长模态；第二，不要只用红色表示错误。推荐使用短文本、轻量图标和固定提示区域。

## 测试与验收

MVP 完成时至少满足：

1. 两个浏览器窗口能通过房间链接进入同一房间。
2. 创建者为黑棋，加入者为白棋，黑棋先手。
3. 双方只能在自己的回合落子。
4. 已有棋子位置不能重复落子。
5. 横、竖、两条斜线五连都能判胜。
6. 胜利后棋局停止，不能继续落子。
7. 获胜五连能被高亮。
8. 刷新页面后可通过 token 回到原座位。
9. 一方断线时，另一方能看到离线状态。
10. 两人准备后能开启下一局。
11. 房间满员时第三人不能加入。
12. 手机浏览器上棋盘可见、可点、无明显遮挡。
13. 360px 宽手机视口下，主要按钮触控区域不小于 48 x 48 CSS px。
14. 键盘 Tab 可以访问首页和房间页所有关键控件，焦点清晰可见。
15. 开启 `prefers-reduced-motion` 后，落子、胜利和弹窗仍可理解。
16. 当前回合、离线和胜负状态不依赖颜色单独表达。

建议测试类型：

- 纯函数单元测试：判胜、坐标校验、房间状态转换。
- Socket 集成测试：创建、加入、落子、重连、再来一局。
- 手动端到端测试：两个浏览器窗口和一台手机访问。
- UI 验收测试：移动端触控命中、焦点状态、减少动态效果、安全边距和文本可读性。

## 部署设计

第一版部署为一个 Node 服务：

1. Vite 构建静态前端。
2. Express 托管构建产物。
3. Socket.IO 挂在同一 HTTP 服务上。
4. 环境变量配置端口和公网 URL。

适合部署到 Render、Railway、Fly.io、VPS 或其他支持 WebSocket 的 Node 平台。若未来拆分前后端，需要补充 CORS、Socket 地址配置和部署健康检查。

## 风险与取舍

- 内存房间状态会在服务重启后丢失。第一版接受该限制，因为目标是演示和验证体验。
- 不登录意味着不能可靠保存长期战绩。第一版只做房间内临时身份。
- 手机端棋盘点击精度是主要体验风险，需要用固定比例棋盘、足够大的落子区域和预落子提示降低误触。
- WebSocket 部署平台必须支持长连接，否则实时对局会不稳定。
- 断线重连依赖本地 token，用户更换浏览器或清除缓存后无法恢复原座位。
- 如果状态栏、弹窗或错误提示遮挡棋盘中心，会破坏对局公平感；所有非关键 UI 必须避开棋盘落子区域。
- 如果只在桌面浏览器验证，手机端大概率出现误触、字号偏小或按钮太密的问题；移动端验收必须进入 MVP 完成标准。

## 后续扩展

优先级较高的第二阶段功能：

1. 悔棋协商。
2. 每步或整局计时。
3. 观战模式。
4. 聊天或快捷表情。
5. 房间规则设置。
6. 数据库保存对局和战绩。
7. 简单账号系统。
8. 随机匹配。
9. AI 练习模式。

这些扩展都应建立在第一版好友房实时对局稳定的基础上。
