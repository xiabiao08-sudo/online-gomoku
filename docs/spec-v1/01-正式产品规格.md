# 棋者弈也 V1 正式产品规格

**文档状态：冻结基线**  
**产品正式名称：棋者弈也**  
**产品说明：好友五子棋 · ONLINE GOMOKU**  
**版本：V1.0**

---

## 1. 产品定义

### 1.1 产品目标

用户无需注册，只需输入昵称、创建棋局并分享链接，即可与一名指定好友完成一局或连续多局 19×19 五子棋。

V1 的优先目标按顺序为：

1. 网站页面稳定打开。
2. 邀请链接能让好友快速进入正确房间。
3. 两名棋手在手机和桌面端均能准确落子。
4. 断线后身份和棋局能够恢复。
5. 第三人及之后用户可以观战并参与实时聊天。

### 1.2 非目标

V1 不做：

- 注册、密码、第三方登录。
- 公共匹配、匹配大厅。
- 排行榜、段位、积分。
- AI、人机对战。
- 永久战绩、棋谱数据库。
- 计时棋、超时判负。
- 三三、四四、长连禁手。
- 观众补位或申请上场。
- 举报、封禁和运营后台。
- 多实例水平扩容。

## 2. 用户角色

### 2.1 创建者

- 创建棋局并获得邀请链接。
- 第一局颜色不固定，由服务端随机决定。
- 具有普通棋手权限，不具有凌驾于另一名棋手之上的“房主裁决权”。

### 2.2 加入棋手

- 第二名进入且未持有观众身份的用户。
- 与创建者共同参与第一局随机分色。

### 2.3 观众

- 两名棋手均已占位后进入的用户。
- 最多 5 人。
- 可以查看棋局、在线状态和实时聊天。
- 可以发送聊天消息。
- 不得落子、申请或审批悔棋、请求再战、结束棋局。
- 不自动补充空缺棋手位。

### 2.4 身份原则

- 每个参与者使用稳定的 `participantId` 和私有 `participantToken` 识别。
- 颜色不是身份。黑白颜色会在再战后交换，参与者 ID 不变。
- Socket 连接 ID 仅代表一次网络连接，不作为永久身份。

## 3. 核心使用闭环

```text
打开首页
→ 输入昵称
→ 创建棋局
→ 复制或系统分享链接
→ 好友打开链接并加入
→ 服务端随机分配黑白
→ 进行对局
→ 产生胜负或和棋
→ 双方选择再来一局
→ 自动交换黑白
→ 继续对局或双方离开
```

### 3.1 首页主操作

主按钮：`创建棋局`  
次操作：`加入棋局`

分享层级：

1. 主操作：`分享房间`，优先调用设备系统分享。
2. 降级操作：复制完整链接。
3. 更多操作：复制房间号、显示二维码。

分享文案：

```text
来「棋者弈也」和我下一局五子棋

19×19 自由五子棋
房间号：8K4D2P
点击进入：<房间链接>
```

## 4. 棋局规则

### 4.1 棋盘

- 19×19 个交叉点。
- 黑棋先手。
- 双方轮流落子。
- 已占用交叉点不可重复落子。

### 4.2 获胜

- 横、竖、左斜或右斜任一方向形成连续 5 颗或以上同色棋子，立即获胜。
- 服务端在每次合法落子后判定。
- 产生胜负后，服务器拒绝所有后续落子。

### 4.3 和棋

- 棋盘 361 个交叉点全部占满且无人获胜，判为和棋。

### 4.4 第一局分色

第二名棋手加入后：

1. 服务端随机分配两名棋手的黑白颜色。
2. 房间进入 `STARTING` 状态。
3. 所有客户端显示约 1 秒：`正在随机分配黑白……`。
4. 显示结果：`你执黑/白，黑棋先行`。
5. 到达服务端公布的开始时间后进入 `PLAYING`。

客户端不能自行随机，也不能在 `STARTING` 状态落子。

### 4.5 再来一局

- 仅在棋局结束后可请求。
- 两名棋手均确认后：
  - 交换黑白颜色。
  - 清空棋盘。
  - 黑棋先手。
  - 清空悔棋和再战请求。
  - 房间短暂进入 `STARTING`，显示“双方已交换黑白”。
- 单方确认时显示“等待对方确认”。

---

## 5. 悔棋规则

操作名称统一为：**申请撤回上一手**。

规则：

1. 只撤回棋盘上的最新一颗棋。
2. 任一棋手均可申请，不要求最新一手属于申请者。
3. 必须由另一名棋手同意。
4. 同意后删除最新棋子，并将回合交还给该棋子的落子方。
5. 申请次数不限。
6. 同一时间只能存在一个悔棋申请。
7. 悔棋申请存在期间暂停落子。
8. 申请者可取消；另一方可同意或拒绝。
9. 任一棋手离线时不能新申请或审批。
10. 棋局结束后不能悔棋。
11. 观众不能参与悔棋流程。

## 6. 在线、断线与离开

### 6.1 在线状态

棋手存在三种参与状态：

- `ONLINE`：当前有有效连接。
- `DISCONNECTED`：网络断开或页面异常关闭，席位保留。
- `LEFT`：用户明确点击离开；席位仍绑定原 token，不给观众或陌生用户补位。

### 6.2 断线

- 任一棋手在对局中断线，房间进入暂停状态。
- 另一名棋手不能继续落子。
- 原棋手使用本机 token 重连后恢复席位。
- 观众不能替补。
- 观众断线只影响其本人。

### 6.3 明确离开

棋手点击离开时显示确认：

```text
确定离开棋局？
离开后当前对局将暂停；该席位不会由观众补充。
若双方都离开，房间会立即结束。
```

删除规则：

- 两名棋手都处于 `LEFT`：立即删除房间并通知全部观众。
- 一名棋手 `LEFT`、另一名仍在：保留房间；剩余棋手可等待或点击“结束房间”。
- “结束房间”视为剩余棋手也明确离开，随后立即删除。
- Socket 异常断开不等同于 `LEFT`。
- 技术兜底：所有参与者长期离线且无人明确操作时，可由服务端按闲置 TTL 清理，建议 30 分钟。
- 服务端重启后，内存房间全部消失，这是 V1 已接受限制。

---

## 7. 观众规则

- 房间棋手位满后，后续用户自动作为观众加入。
- 最多 5 名观众；第 6 名观众收到 `SPECTATOR_CAPACITY_REACHED`。
- 观众有独立 token，可刷新后恢复观众身份。
- 观众不显示历史聊天。
- 观众列表只显示当前在线或暂时断线的观众昵称与状态。
- 棋手明确离开后，观众仍不补位。

---

## 8. 聊天规则

### 8.1 产品规则

- 棋手和观众均可发送消息。
- 新加入、刷新或重新连接后，不显示过去聊天内容。
- 服务端不把聊天记录放入房间快照，也不写入数据库。
- 聊天只作为在线事件广播给当时已连接的参与者。

### 8.2 限制

- 单条消息去除首尾空格并折叠连续空白。
- 最大 160 个字符。
- 每个参与者至少间隔 2 秒才能再次发送。
- 空消息拒绝。
- 参与者离开或 token 无效时拒绝。
- 手机端聊天默认折叠，显示未读数。

---

## 9. 移动端落子交互

### 9.1 默认设置

- 手机端默认开启“确认落子”。
- 桌面端可默认关闭，也允许用户开启。
- 设置保存在当前浏览器的本地存储中，下次继续使用。

### 9.2 确认模式开启

```text
第一次轻点空交叉点
→ 生成半透明候选棋子
→ 显示坐标和 5×5 局部放大镜
→ 再次点击同一点或点击“确认落子”
→ 向服务端提交
```

### 9.3 确认模式关闭

- 单次合法轻点直接提交。
- 拖动和缩放手势结束时不得误提交。

### 9.4 手势

- 单指轻点：选点。
- 单指移动超过 8px：平移棋盘，不落子。
- 双指：缩放棋盘，不落子。
- 双指开始时取消候选落子。
- `pointercancel`、触点数量变化或缩放结束时清除点击判定。
- 缩放范围：100%、125%、150%、175%、200%。
- 对手最后一步不在当前视口时显示“查看最后一步”，不强制移动玩家视角。
- 提供“棋盘居中”和“恢复 100%”。

## 10. 对局页面信息架构

### 10.1 顶部固定状态栏

优先显示：

1. 是否轮到自己。
2. 自己执黑或执白。
3. 对手在线、断线或已离开。
4. `确认落子：开/关`。

示例：

```text
● 你执黑 ｜ 轮到你 ｜ 对手在线
确认落子：开
```

### 10.2 主区域

- 房间号和分享入口。
- 19×19 棋盘。
- 候选坐标和放大镜。
- 观众人数简要提示。

### 10.3 底部固定操作栏

有候选点：

```text
取消选择 ｜ 确认落子 ｜ 聊天 2 ｜ 更多
```

无候选点：

```text
确认：开 ｜ 悔棋 ｜ 聊天 2 ｜ 更多
```

“更多”包含：

- 放大、缩小。
- 回到最后一步。
- 棋盘居中。
- 分享房间。
- 请求再来一局。
- 离开棋局。

### 10.4 聊天

- 桌面端：右侧辅助栏。
- 手机端：底部抽屉，最大高度建议 60vh。
- 有新消息但抽屉关闭时显示未读角标。

## 11. 胜负反馈

顺序：

1. 服务端确认胜负或和棋。
2. 客户端禁止落子。
3. 高亮完整获胜连线。
4. 棋盘中央显示“黑胜”“白胜”或“和棋”印章。
5. 约 600ms 后打开底部结果卡。
6. 棋盘保持可查看、平移和缩放。

结果卡按钮：

- `再来一局`
- `查看棋盘`
- `离开棋局`

不使用遮挡棋盘的大型粒子、烟花或强制全屏弹窗。

## 12. 连接与加载状态

必须区分：

- `BOOTING`：前端资源初始化。
- `CONNECTING`：首次连接后端。
- `WAKING_SERVER`：连接时间较长，可能正在唤醒免费后端。
- `JOINING_ROOM`：Socket 已连接，正在加入或恢复房间。
- `CONNECTED`：房间同步完成。
- `RECONNECTING`：连接中断，自动重连。
- `ROOM_NOT_FOUND`：房间不存在或服务端重启后失效。
- `ROOM_CLOSED`：双方明确离开，房间已结束。
- `SERVER_UNAVAILABLE`：多次连接失败。

加载动画不显示虚假百分比，也不承诺具体时间。

## 13. 错误提示规范

错误码与用户文案分离。客户端根据错误码显示中文：

| 错误码 | 用户文案 |
|---|---|
| `ROOM_NOT_FOUND` | 棋局不存在或已经结束。 |
| `SPECTATOR_CAPACITY_REACHED` | 观众席已满。 |
| `INVALID_TOKEN` | 无法恢复原身份，请重新进入。 |
| `NOT_YOUR_TURN` | 还没有轮到你。 |
| `CELL_OCCUPIED` | 该位置已有棋子。 |
| `PLAYER_OFFLINE` | 对方暂时离线，棋局已暂停。 |
| `GAME_NOT_PLAYING` | 当前不能落子。 |
| `UNDO_ALREADY_PENDING` | 已有待处理的悔棋申请。 |
| `UNDO_NOT_AVAILABLE` | 当前不能悔棋。 |
| `CHAT_RATE_LIMITED` | 消息发送太快，请稍后再试。 |
| `ROOM_CLOSED` | 双方已离开，本棋局已经关闭。 |

## 14. 非功能要求

### 14.1 权威性

- 棋盘、回合、颜色、胜负、悔棋、再战和身份均由服务端决定。
- 客户端只提交意图，不提交最终状态。

### 14.2 幂等性

所有改变状态的客户端操作包含 `actionId`。服务端对同一参与者的重复 `actionId` 返回首次结果，不重复执行，用于防止重试造成重复落子或重复聊天。

### 14.3 安全边界

- Socket CORS 仅允许正式前端来源和本地开发来源。
- token 不放入 URL。
- 昵称最长 16 字符。
- 服务端校验所有坐标、角色、回合和房间状态。
- 观众权限必须由服务端拒绝，而非只隐藏按钮。

### 14.4 可用性

- 前端静态页面与游戏后端拆分。
- 后端未唤醒时，前端仍可展示品牌、规则和连接状态。
- 后端单实例运行，避免内存房间在不同实例之间分裂。

## 15. V1 验收标准

### 创建与加入

- [ ] 创建者输入有效昵称可创建棋局。
- [ ] 分享链接可直接进入对应房间。
- [ ] 第二名棋手加入后由服务端随机分色。
- [ ] 两名棋手看到相同分色、棋盘和开始时间。
- [ ] 第三至第七名用户作为观众加入。
- [ ] 第八名用户收到观众席已满提示。

### 棋局

- [ ] 仅当前回合棋手可落子。
- [ ] 五子及以上立即结束。
- [ ] 满盘无胜者判和。
- [ ] 结束后无法落子和悔棋。
- [ ] 再战双方确认后交换颜色。

### 悔棋

- [ ] 任一棋手可申请撤回最新一手。
- [ ] 对方同意后仅撤回一颗。
- [ ] 回合恢复为被撤回棋子的颜色。
- [ ] 拒绝或取消不会改变棋盘。

### 断线与离开

- [ ] 棋手断线后暂停。
- [ ] 原 token 可恢复棋手身份。
- [ ] 观众不补位。
- [ ] 双方明确离开后房间立即关闭。
- [ ] 服务端重启后旧房间显示失效。

### 聊天

- [ ] 棋手和观众均可发送。
- [ ] 新加入、刷新或重连后无历史消息。
- [ ] 2 秒发送间隔和 160 字符限制生效。

### 手机交互

- [ ] 默认开启确认落子并记住设置。
- [ ] 候选棋、坐标和放大镜一致。
- [ ] 拖动和双指缩放不会落子。
- [ ] 100%–200% 缩放正常。
- [ ] 顶部状态和底部操作在棋盘浏览时可见。
