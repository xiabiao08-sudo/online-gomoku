# 棋者弈也 V1 前后端拆分部署配置

目标：同一个 GitHub 仓库建立两个 Render 服务。

```text
Render Static Site
└── React / Vite 前端，立即加载页面和连接动画

Render Free Web Service
└── Express + Socket.IO，房间状态保存在单实例内存
```

---

## 1. 为什么拆分

单体 Web Service 休眠或启动失败时，前端 HTML 也无法加载，用户只能看到平台错误页。拆分后：

- 静态前端由 CDN 提供，可以先显示品牌、规则和“正在连接游戏服务器”。
- 只有 Socket.IO 后端需要从免费实例休眠中恢复。
- 前端与后端可以独立部署和回滚。
- 前端 `/room/:roomId` 使用 SPA rewrite，直接打开邀请链接仍能进入应用。

---

## 2. 代码结构

V1 可继续使用当前单仓库结构：

```text
online-gomoku/
├── src/                 # React 前端及 shared 类型
├── server/              # Express + Socket.IO
├── tests/
├── e2e/
├── package.json
├── vite.config.ts
└── render.yaml
```

必须修改：

1. 前端 Socket 客户端从 `VITE_SOCKET_URL` 读取后端地址。
2. 后端不再负责生产环境静态文件路由。
3. 后端 CORS 使用 `FRONTEND_ORIGINS` 白名单。
4. `package.json` 区分前端构建和后端启动脚本。

## 3. 前端 Socket 配置

`src/client/socketClient.ts`：

```ts
import { io } from "socket.io-client";

const socketUrl = import.meta.env.VITE_SOCKET_URL;

if (!socketUrl) {
  throw new Error("VITE_SOCKET_URL is required");
}

export const socket = io(socketUrl, {
  autoConnect: true,
  transports: ["websocket", "polling"],
  reconnection: true,
  reconnectionAttempts: Infinity,
  reconnectionDelay: 1000,
  reconnectionDelayMax: 10000,
  randomizationFactor: 0.5,
  timeout: 20000,
  ackTimeout: 8000,
  retries: 2
});
```

注意：

- `VITE_*` 会进入浏览器构建产物，只能放公开配置，不能放密钥。
- 使用有限动作重试时，服务端必须按 `actionId` 去重。
- 页面重连后始终请求完整房间快照。

## 4. 后端 CORS

`server/index.ts`：

```ts
const allowedOrigins = new Set(
  (process.env.FRONTEND_ORIGINS ?? "http://localhost:5173")
    .split(",")
    .map((value) => value.trim())
    .filter(Boolean)
);

const io = new Server(httpServer, {
  cors: {
    origin(origin, callback) {
      if (!origin || allowedOrigins.has(origin)) {
        callback(null, true);
        return;
      }
      callback(new Error("ORIGIN_NOT_ALLOWED"));
    },
    methods: ["GET", "POST"]
  }
});
```

后端只保留：

```ts
app.get("/health", (_req, res) => {
  res.status(200).json({ ok: true });
});
```

删除生产静态文件部分：

```ts
app.use(express.static(distPath));
app.get("*", ...);
```

## 5. 推荐 package.json 脚本

```json
{
  "scripts": {
    "dev:client": "vite",
    "dev:server": "tsx watch server/index.ts",
    "build:client": "vite build",
    "start:server": "tsx server/index.ts",
    "typecheck": "tsc --noEmit",
    "test": "vitest run",
    "test:e2e": "playwright test",
    "check": "npm run typecheck && npm run test && npm run build:client"
  }
}
```

V1 保留 `tsx` 启动服务端，减少构建改造范围。后续需要进一步硬化时，再增加独立服务端编译产物。

## 6. Render Blueprint

仓库根目录使用本规格包的 `render.split.yaml`，最终可命名为 `render.yaml`。

核心配置：

```yaml
services:
  - type: web
    name: qizheyiy-gomoku-web
    runtime: static
    buildCommand: npm ci && npm run check
    staticPublishPath: ./dist
    routes:
      - type: rewrite
        source: /*
        destination: /index.html

  - type: web
    name: qizheyiy-gomoku-api
    runtime: node
    plan: free
    buildCommand: npm ci && npm run typecheck && npm run test
    startCommand: npm run start:server
    healthCheckPath: /health
    numInstances: 1
```

完整配置见 `render.split.yaml`。

## 8. SPA 路由

静态站点必须配置：

```yaml
routes:
  - type: rewrite
    source: /*
    destination: /index.html
```

作用：直接打开或刷新 `/room/8K4D2P` 时，Render 返回 `index.html`，再由 React 解析路径。

客户端还必须监听 `popstate`，保证浏览器返回和前进后 UI 与 URL 一致。

## 9. 健康检查与端口

后端：

- 监听 `process.env.PORT`。
- Host 为 `0.0.0.0`。
- `/health` 只执行轻量就绪检查并返回 2xx。
- 单实例 `numInstances: 1`，因为房间在内存中。

示例：

```ts
const port = Number(process.env.PORT ?? 8788);
const host = process.env.HOST ?? "0.0.0.0";
httpServer.listen(port, host);
```

## 10. 优雅关机

部署时 WebSocket 连接可能被中断。服务端接收 `SIGTERM` 后：

```ts
async function shutdown(signal: string) {
  io.emit("server:shutdown", { signal });

  io.close(() => {
    httpServer.close(() => process.exit(0));
  });

  setTimeout(() => process.exit(1), 25_000).unref();
}

process.once("SIGTERM", () => void shutdown("SIGTERM"));
process.once("SIGINT", () => void shutdown("SIGINT"));
```

客户端收到 `server:shutdown` 后显示“服务器正在更新，正在重新连接”。

## 11. 部署顺序

### 方案 A：Blueprint

1. 提交代码和 `render.yaml`。
2. Render Dashboard 创建 Blueprint。
3. 确认两个服务创建成功。
4. 若自动引用的默认 URL 不符合预期，在服务 Environment 中修改：
   - 前端 `VITE_SOCKET_URL`
   - 后端 `FRONTEND_ORIGINS`
5. 重新部署前端和后端。

### 方案 B：手工创建，排错更直观

1. 先创建后端 Web Service。
2. 取得后端 HTTPS URL。
3. 创建 Static Site，设置 `VITE_SOCKET_URL`。
4. 取得前端 URL。
5. 在后端设置 `FRONTEND_ORIGINS` 并重新部署。
6. 配置静态站点 rewrite。

## 13. 当前仓库到目标配置的修改清单

| 当前 | 目标 |
|---|---|
| 前后端由一个 Web Service 提供 | Static Site + Web Service |
| Socket 客户端默认同源 | 使用 `VITE_SOCKET_URL` |
| Socket CORS `origin: "*"` | 精确 Origin 白名单 |
| Express 提供 `dist` | 后端仅 API 和 Socket.IO |
| 创建者固定黑、加入者固定白 | 第一局随机，之后交换 |
| 第三人被拒绝 | 最多 5 名观众 |
| 房间快照保留 50 条聊天 | 聊天实时广播且无历史 |
| 再战清盘但不换色 | 双方确认后交换颜色 |
