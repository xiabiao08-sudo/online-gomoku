# 在线联机五子棋复盘

日期：2026-07-06

## 结论

这次项目的有效路线是：先锁定好友房 MVP，再用单体 Node 服务承载 React 静态资源和 Socket.IO 实时后端，最后通过测试、局域网分享、公网隧道和视觉截图逐步验收。这个路线适合复用到大多数轻量网页联机小游戏，尤其是棋类、牌类、回合制对战和小型房间制游戏。

## 最小可行产品边界

本次做对的边界：

- 不登录，只用昵称和本地 token 恢复身份。
- 只做好友房，不做匹配大厅。
- 只支持两人，不做观战和多人扩展。
- 规则使用自由五子棋，不做禁手。
- 房间状态保存在服务端内存，不接数据库。
- 第一版先保证创建、加入、落子、判胜、重连、再来一局闭环。

这个边界让项目能在较短时间内成品化，同时仍然保留真实联机体验。

## 核心架构

采用单体实时网页应用：

- `server/index.ts`：Express + HTTP server + Socket.IO，提供健康检查、静态资源和实时通信。
- `server/roomStore.ts`：内存房间、玩家、棋盘和回合状态。
- `server/socketHandlers.ts`：Socket 事件入口，统一 ack 返回。
- `src/shared/game.ts`：纯游戏规则和类型，便于单元测试。
- `src/client/socketClient.ts`：前端 Socket.IO 客户端封装。
- `src/components/`：首页、房间页、棋盘、玩家面板、状态提示。

关键原则：服务端是最终裁判，前端只发送玩家意图。落子是否合法、当前回合、胜负判定都由服务端决定。

## 可复用事件模型

五子棋使用的事件可以泛化为大多数房间制联机小游戏：

| 事件 | 用途 |
| --- | --- |
| `room:create` | 创建房间并分配第一个玩家 |
| `room:join` | 昵称加入或 token 重连 |
| `room:state` | 服务端广播完整公共房间状态 |
| `game:placeStone` | 玩家提交游戏动作 |
| `game:restartReady` | 双方准备下一局 |

后续做其他游戏时，把 `game:placeStone` 替换为更通用的 `game:action` 或领域动作即可，例如 `game:playCard`、`game:movePiece`、`game:submitGuess`。

## 分享与部署经验

本次踩到的核心问题：

- `127.0.0.1` 只能本机访问，不能分享给别人。
- `192.168.x.x` / `10.x.x.x` 只能同一局域网访问。
- 异地朋友需要公网 HTTPS 域名，临时可用 Cloudflare Tunnel，长期应部署到 Render/Railway/Fly.io/VPS。
- Socket.IO 需要支持 WebSocket 的 Node Web Service，不能只部署成静态站点。
- 内存房间状态只适合单实例，多实例需要 Redis Adapter 或持久化状态。

这类项目应从第一天就提供：

- `/health` 健康检查。
- `npm start` 部署启动命令。
- `HOST=0.0.0.0` 或默认监听全网卡。
- README 中明确本机、局域网、公网三种地址的区别。

## UI 与交互经验

五子棋暴露出一个通用游戏 UI 风险：视觉网格和交互坐标必须使用同一坐标系。

本次错误是棋盘线按 `14` 等分绘制，但棋子按 `15 x 15` 网格居中，导致棋子没有落在交点上。修复方式是让棋盘线绘制在 15 个点击点的中心坐标上，并用浏览器量测中心偏差。

后续做棋类或格子游戏时必须验收：

- 棋子、格线、点击热区是否同坐标。
- 最近一步和胜利高亮是否不遮挡棋子。
- 手机视口下触控区是否足够大。
- 状态栏、按钮、弹窗是否避开主操作区域。
- 桌面和移动端都截图检查，不只看功能测试。

## 测试组合

这次有效的测试层次：

- `tests/game.test.ts`：纯规则，如棋盘、落子、判胜。
- `tests/roomStore.test.ts`：房间状态转换，如加入、重连、再来一局。
- `tests/socket.test.ts`：实时事件与 ack 行为。
- `e2e/gomoku.spec.ts`：两个浏览器创建/加入/交替落子/胜利。
- Playwright 截图与 DOM 几何量测：验证视觉坐标和响应式布局。

经验：游戏项目不能只跑单元测试。很多关键问题只会出现在真实浏览器、双客户端、移动视口或截图里。

## 推荐后续演进顺序

1. 正式部署到 Render/Railway 单实例，拿稳定公网域名。
2. 增加房间规则配置，例如棋盘尺寸、先手、是否允许悔棋。
3. 增加观战和房间列表，但仍保持游戏状态服务端裁判。
4. 接入 Redis Adapter 或数据库，支持重启恢复和多实例。
5. 抽象通用房间框架，复用于其他小游戏。

## 下次创建类似游戏的启动清单

1. 写清 MVP 边界：玩家数、房间方式、是否登录、是否保存历史。
2. 先实现纯规则模块和测试。
3. 再实现服务端房间状态机。
4. 用 Socket.IO ack 处理客户端动作结果。
5. 前端只订阅完整公共状态，不拼补丁。
6. 加入 `/health`、`npm start`、公网部署说明。
7. 验证本机、局域网、公网三种访问路径。
8. 用双浏览器 E2E 验证真实联机。
9. 用截图和几何量测验证主要 UI 坐标。
10. 把复盘写入项目文档和可复用 Skill。
